# Eudaemon Alpha Initiative

**Institutional Intelligence for the Nostra-Cortex Ecosystem**
**Strategic Architecture Report - Refreshed 2026-03-12**

## Executive Summary

Eudaemon Alpha remains the institutional research-agent initiative for the Nostra/Cortex ecosystem, but its architectural framing is now tightened around the primitives Cortex already implements.

The important correction is this:
- Eudaemon does not need a new workspace primitive.
- Eudaemon does not need a bespoke task model.
- Eudaemon does not treat initiative plans as runtime plans.

Instead, the current stage architecture is:
- heap blocks for exploratory notes and working material
- closeout or workflow tasks for operational follow-through
- workflow artifacts for executable plans
- research initiatives and DPub artifacts for governed lineage

## Strategic Position

Eudaemon is the institutional integration point between:
- Initiative 124: heap workspace runtime
- Initiative 126: agent authority and lifecycle controls
- Initiative 127: sandboxed repo/context boundaries
- Initiative 130: space capability activation and compiled plans
- Initiative 133: evaluation and grading enrichments
- Initiative 134: governed workflow authority and execution

This means Initiative 132 should continue by binding to these surfaces, not by modeling everything as generalized artifacts or external branch workflows.

## Current Stage Principles

1. Nostra defines institutional identity, initiatives, DPub lineage, and governed contributions.
2. Cortex defines the active work surface, execution events, action affordances, and workflow runtime.
3. Exploratory work belongs on the heap and is promoted when it becomes durable.
4. Task progress belongs on closeout/workflow surfaces, not inside notes or initiatives.
5. Knowledge references guide architecture but do not outrank local contracts.

## Operational Shape

### Workspace
- Space-scoped and capability-governed
- Heap-enabled for working notes, charts, pointers, and A2UI payloads
- DPub-enabled for chronicle publication

### Working loop
1. Observe graph, space, and sandbox signals
2. Pull selected heap context
3. Analyze within recommendation-only authority
4. Emit exploratory heap material and lifecycle evidence
5. Promote mature work into governed artifacts when needed

### Promotion paths
- Heap note -> proposal
- Heap note -> DPub update
- Heap synthesis -> workflow draft
- Workflow draft -> definition -> instance

## What Was Removed

The following assumptions are no longer treated as primary architecture:
- artifact-only scratchpad modeling as the main workspace
- git-branch submission as the primary proposal path
- older canonical-workflow-engine claims that bypass Initiative 134
- separate Eudaemon workspace/task primitives outside the current Cortex stack

## Current Recommendation

Continue Initiative 132 only against the current work primitive split documented in:
- `PLAN.md`
- `WORK_PRIMITIVES_ARCHITECTURE.md`
- `README.md`

That keeps the initiative aligned to the present state of Cortex instead of continuing from stale pre-heap and pre-134 assumptions.
